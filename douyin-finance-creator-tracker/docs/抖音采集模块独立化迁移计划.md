# 抖音采集模块独立化迁移计划

## 1. 目标

将 Fund Insight Desk 当前依赖 MediaCrawler 子进程和中间 SQLite 的同步链路，迁移为后端内部的抖音采集模块。

迁移后的采集模块直接返回博主资料与作品原始数据，由业务服务直接写入 `fund_insight.db`。不再通过 `database/sqlite_tables.db`、stdout 日志正则或 `creator_hash` 跨库匹配来传递作品数据。

目标链路：

```text
定时器 / 手动同步
  -> FundInsightService
  -> DouyinCollector.fetch_creator(sec_user_id, since)
  -> CreatorRaw + list[AwemeRaw]
  -> 业务映射与幂等写入 fund_insight.db
  -> 同步状态与前端轮询
```

## 2. 当前链路

当前实现位于 `backend/app.py`：

1. `run_sync()` 为每个博主启动一个 MediaCrawler 子进程。
2. 子进程以 `--type creator` 调用抖音采集器。
3. MediaCrawler 将作品详情写入根项目的 `database/sqlite_tables.db` 中的 `douyin_aweme` 表。
4. 后端从子进程日志中提取 `creator_hash`、昵称、头像和时长。
5. 后端通过 `creator_hash` 和发布时间从 `douyin_aweme` 读取作品，再导入 `fund_insight.db`。

当前问题：

- 子进程、日志解析和双 SQLite 增加故障点。
- 作品数据要经过“MediaCrawler 库 -> 业务库”的二次筛选。
- `creator_hash` 是跨库关联的间接标识，异常时会拒绝导入。
- 每个博主需要等待子进程完整退出后才能导入。
- 调度器排好的同步顺序会在 `SELECT ... WHERE id IN (...)` 的再次查询中丢失。

## 3. 目标模块结构

在 `backend/crawler/douyin/` 下建立独立模块：

```text
crawler/douyin/
  __init__.py
  models.py          # CreatorRaw、AwemeRaw、CollectResult
  collector.py       # 采集编排，返回结构化结果
  client.py          # 抖音用户、作品列表、作品详情 API
  session.py         # Playwright/CDP、Cookie、登录态生命周期
  signer.py          # 请求参数签名封装
  parser.py          # sec_user_id 与原始响应字段解析
  errors.py          # LoginExpired、RateLimited、DataFetchError
```

业务层保留：

```text
backend/service.py   # 写入 creators/videos、幂等更新、查询接口
backend/app.py       # API、同步状态、后台线程
backend/intraday_scheduler.py # 调度时间与博主排序
```

## 4. 数据契约

采集层不得访问业务数据库，也不得写入中间 SQLite。它只返回原始结果。

建议数据模型：

```python
@dataclass
class CreatorRaw:
    sec_user_id: str
    platform_uid: str
    nickname: str
    avatar_url: str

@dataclass
class AwemeRaw:
    aweme_id: str
    create_time: int
    desc: str
    cover_url: str
    playback_url: str
    duration_seconds: int
    raw_payload: dict[str, Any]

@dataclass
class CollectResult:
    creator: CreatorRaw
    awemes: list[AwemeRaw]
    reached_cursor: bool
```

业务库建议给 `videos` 增加 `raw_payload_json TEXT`，保存作品详情响应的原始 JSON。页面仍使用已映射字段；原始数据仅用于排障、字段补充和重新解析。

## 5. 分阶段实施

### 阶段 0：许可证与范围确认

- 当前 MediaCrawler 源文件标注为 `NON-COMMERCIAL LEARNING LICENSE 1.1`。
- 商业使用前必须取得原作者授权，或采用可以合法使用的独立实现。
- 不复制整个 MediaCrawler 项目；仅在确认授权范围后，基于必要逻辑实现独立模块并保留来源与许可证声明。

验收：确认许可、部署用途与可复制代码范围。

### 阶段 1：建立采集接口和数据模型

- 创建 `models.py`、`errors.py` 与 `collector.py` 的接口。
- 为 `DouyinCollector` 定义 `start()`、`fetch_creator()`、`close()` 生命周期。
- 暂不接入真实请求，可使用固定测试数据。
- 为 `CollectResult` 和字段映射写单元测试。

验收：业务层可通过接口消费 `CollectResult`，不依赖 MediaCrawler 表结构。

### 阶段 2：迁移长期 Chrome 与登录态

- 将现有 CDP 连接逻辑封装到 `session.py`。
- 优先连接启动脚本拉起的长期 Chrome；连接失败时给出明确登录态错误。
- 一个浏览器上下文内按需创建 Page；每次博主采集结束关闭该 Page，不关闭长期 Chrome。
- 登录、扫码和 Cookie 更新必须由会话层负责，不能散落在业务层。

验收：应用重启后可复用浏览器登录态；多次同步不会启动额外 Chrome 进程。

### 阶段 3：迁移抖音请求和作品采集

- 实现用户资料请求，获取昵称、头像和平台 UID。
- 实现创作者作品列表分页，请求参数携带 `max_cursor`。
- 实现时间游标：只收集 `create_time > since` 的作品；遇到已存在时间点停止继续翻页。
- 对列表得到的每个 `aweme_id` 请求详情，受限并发并加入固定限速。
- 从详情响应提取封面、播放地址、时长、描述及完整原始响应。

验收：针对一个已登录博主，采集结果中的作品数量、作品 ID、标题、发布时间与现有 MediaCrawler 结果一致或可解释差异。

### 阶段 4：直接写入业务库

- 新增 `FundInsightService.upsert_collected_creator(result, creator_id)`。
- 使用 `external_id = aweme_id` 做幂等更新。
- 直接更新 `creators.name`、`avatar`、`last_synced`、`last_crawled_at`。
- 新作品写入 `videos`；已有作品更新封面、播放地址、描述和原始数据。
- 只有单个博主的采集和落库都成功后才更新该博主 `last_synced`。

验收：关闭 MediaCrawler SQLite 后仍能完成首次同步、重复同步和新增作品同步。

### 阶段 5：并行验证与切换

- 增加配置开关：`SYNC_COLLECTOR=legacy|direct`，默认 `legacy`。
- 对同一测试博主分别运行两条链路，比较 aweme ID、发布时间、标题、封面和播放地址。
- 记录差异，不以“新增数量”作为唯一指标，因为已有 ID 会触发更新而非新增。
- 验证稳定后切换默认值到 `direct`，保留旧链路一个发布周期作为回滚通道。

验收：连续多个定时周期无登录异常、无重复作品、无漏入库的新增作品。

### 阶段 6：清理旧链路

- 删除 `run_crawler_process()`、`crawler_command()`、日志正则和 `media_rows_for_creator()`。
- 删除对 `MEDIA_CRAWLER_ROOT`、`MEDIA_CRAWLER_DB`、`creator_hash` 跨库导入的依赖。
- 仅在确认不再需要历史兼容时，移除 `crawler_creator_hash` 字段及旧的修复逻辑。
- 从启动脚本中移除 MediaCrawler 专用启动步骤；保留长期 CDP Chrome 启动。

验收：Fund Insight Desk 单独部署，后端目录外没有 MediaCrawler 运行时依赖。

## 6. 同步顺序与失败策略

- 全量定时同步按 `last_synced` 升序，未同步的博主优先。
- 同步任务中必须保留传入的已排序博主 ID，不能用无排序的 `IN (...)` 查询重排。
- 单个博主失败时记录该博主的 `source_status` 和错误消息，继续处理后续博主。
- 登录过期属于全局依赖失败，可以停止任务并要求用户重新认证。
- 详情请求失败仅影响对应作品；记录失败 aweme ID，不能让其余作品丢失。

## 7. 验收清单

- 首次同步：最新 15 条候选中，近 3 天作品按规则入库。
- 增量同步：只新增或更新游标后的作品，不产生重复 `external_id`。
- 数据完整性：昵称、头像、标题、描述、封面、播放 URL、时长可用。
- 页面行为：同步状态能显示当前博主；完成后 dashboard 与时间线自动刷新。
- 浏览器行为：长期 Chrome 只启动一次；博主采集 Page 在每次采集结束后关闭。
- 容错：一个博主失败不会阻断其他博主；登录失效有明确可操作错误。
- 部署：删除或停止 MediaCrawler SQLite 后，direct 模式仍可工作。

## 8. 回滚方案

在阶段 5 前保留 `legacy` 同步链路。若 direct 模式发生请求签名、登录态或作品解析异常：

1. 将 `SYNC_COLLECTOR` 切回 `legacy`。
2. 保留 direct 模式的错误日志和原始响应样本。
3. 修复后先对单一测试博主验证，再恢复全量定时同步。
